# UXポリシー

## 目的

先生と生徒が授業中に迷わず、安全に、物理現象の観察へ集中できる体験を定める。本書は画面、操作、案内、学習体験の正本である。システム責務は [architecture.md](architecture.md)、配置は [repository-structure.md](repository-structure.md) を参照する。

## 最上位原則

1. 初見の利用者が30秒以内に、何を見るか、何を押すか、何が起きるかを理解できる。
2. 説明を読むだけでなく、実際の操作と観察で理解する。
3. 授業で最もよく使う観察を初期状態にする。
4. 物理現象と計測結果を主役にし、設定や説明で画面を埋めない。
5. 失敗時には評価ではなく、次に取る行動を示す。

## 教育体験

- ページ名だけで計測対象が分かるようにする。
- 軸、単位、正方向、値の意味を画面内に示す。
- 操作と結果の因果関係がすぐ分かるよう、入力後ただちに状態または値を更新する。
- 主要な実験の流れを、開始、操作、観察、停止、読み取りの順で体験させる。
- 初期値は授業で一般的な手順に合わせ、例外的な用途はタブや調整カードへ分ける。
- 正解だけを求めず、予想、実測、再挑戦の流れを用意する。
- 追加課題は必須手順から分離し、「もっとやる」で選べるようにする。

## 画面構成

- スマホ・タブレットの縦向きを基本とする。
- 横向きかつタッチ端末では、共通の縦向きガードを全面表示する。
- マウス利用を前提とする横長PCでは、縦向きガードを出さない。
- ヘッダーは高さ52pxを基準とし、黄色 `#ffe000` と暗色文字で統一する。
- ヘッダーには戻る操作、ページ名、主要操作、チュートリアルを必要な範囲で置く。
- 本文の最大幅を抑え、タブレットで視線が左右に散らないようにする。
- 計測値は `font-variant-numeric: tabular-nums` を使い、桁の変化で表示位置を揺らさない。
- 関連する調整項目は一つのカードへまとめる。
- グラフ見出しはデータを隠さない空き領域へ置き、縦の余白を増やしすぎない。

## 色とアイコン

- テーマ色は黄色 `#ffe000` とする。
- 3軸の基本色は X＝青 `#4C8DF0`、Y＝緑 `#4FC96B`、Z＝黄 `#F2C744` とする。
- 色だけに意味を持たせず、x・y・zや状態文を併記する。
- 操作アイコンは原則として、塗りが単純なフラットSVGを使う。
- 同じ操作には全ページで同じ図形を使う。
- アイコンボタンには `title` を付ける。
- 意味を持つSVGには `aria-label` を付ける。
- チュートリアル文中の操作アイコンは `TUT_ICONS` を使う。

## タップと主要操作

- 主要なボタンは指で確実に押せる余白を持たせる。
- ヘッダー操作を装飾やオーバーレイより前面に置く。
- タップ可能な場所は、文言、カーソル、強調のいずれかで分かるようにする。
- 通常表示のグラフタップは拡大とする。
- 拡大表示では線タップを値の読み取り、線から離れたタップを縮小とする。
- 画面の状態に合わせ、ヒントも「拡大」から「値・縮小」へ切り替える。
- 値を選んだときは、グラフ内の吹き出しと見出しの値を更新する。
- タップ値は有効数字3桁を基準にする。
- ピンチやパンを備える場合も、単純なタップ操作を壊さない。

## 状態表示

- 待機中、計測中、一時停止、完了、中断、エラーを区別する。
- ボタンの文言とアイコンは現在の動作を表す。
- 計測開始前には、次に何を待っているかを表示する。
- 一時停止後には、再開方法を表示する。
- リセット後は初期状態の案内へ戻す。
- 長時間記録を自動停止する場合は、停止理由と回復操作を表示する。
- 値がない状態は `--` など、数値と誤認しない表現にする。

## 権限と端末差

- センサー・マイクの権限要求は、利用者が開始を押した直後に行う。
- 権限が必要な理由を、操作の文脈から理解できるようにする。
- OSやブラウザで許可ダイアログの文言が異なるため、特定の全文に依存せず「許可」を押すよう案内する。
- 拒否、API非対応、センサー非搭載、データ未取得を区別できる範囲で案内する。
- エラー文は技術用語だけで終わらせず、利用者が次に確認することを書く。
- 別タブや別アプリへ移ったら計測を止め、戻っても勝手に再開しない。
- マイク計測では、実験値を変える自動ゲイン、ノイズ抑制、エコー除去を原則無効にする。

## 初回オンボーディング

初回オンボーディングは「画面の場所案内」であり、実験の体験チュートリアルとは分ける。

- 初回訪問時だけ自動開始する。
- 最大3ステップとする。
- 各説明は20文字程度を目安にする。
- 最初に主要な結果、次に開始操作を案内する。
- 最終ステップは必ず `#missionBtn` を指し、体験チュートリアルへつなげる。
- 呼称は「ミッション」ではなく「チュートリアル」に統一する。
- 完了またはスキップを `localStorage` に記録し、毎回表示しない。
- ページごとに一意な保存キーを使う。
- 戻る、次へ、閉じる操作が対象要素から離れすぎないよう配置を調整する。

## 体験チュートリアル

- チュートリアルボタンはヘッダーに置き、初回は短いパルスで存在を知らせる。
- 課題は一度に一つだけ表示する。
- 進捗ドットと残り件数を表示する。
- 課題文は「〜しよう」「〜してみよう」の具体的な行動にする。
- 自動判定できる操作は、成功した時点で自動的に進める。
- 観察そのものが目的の場合だけ「見えた！つぎへ」を使う。
- 計測中だけ判定できる課題では、停止中に進めず開始操作を案内する。
- 失敗時は「もう少し長く」「右へ動かしてね」など、一つの修正行動を示す。
- 予想を含む課題は、予想後に実測し、違った場合も次の試行へ導く。
- 表示上の成功と合格判定は同じイベントまたは状態を使う。
- 完了時はアイコンに小さな完了バッジを重ねる。
- 完了後に「最初から」「もっとやる」「閉じる」を提示する。
- 進捗は再読み込み後も保持する。

## 文言

- 中学生が一度で理解できる日本語を使う。
- 一文に複数の操作を詰め込まない。
- ボタン名と案内文で同じ用語を使う。
- 「すぐ」「簡単」など、利用者の状況を決めつける表現を避ける。
- 計測失敗を利用者のせいにしない。
- 単位記号だけでなく、必要に応じて量の名称を併記する。
- 許可案内はOSの版に依存しない表現にする。

## エラーと回復

- エラーは対応するページ内に表示し、画面外のコンソールだけへ出さない。
- 権限拒否では、許可されなかったことを明示する。
- データが来ない場合は、一定時間後にセンサー非搭載などの可能性を示す。
- 中断・リセット・戻る操作で安全な初期状態へ戻れるようにする。
- エラー後もページ全体を再読み込みせず、可能な範囲で再試行できるようにする。
- 利用者が失うデータを伴う操作には、ボタンの `title` で結果を示す。

## レビュー時の問い

1. 初見で主要な値と開始操作が分かるか。
2. 30秒以内に体験チュートリアルへ到達できるか。
3. 縦向きのスマホ・タブレットで、主要操作が一画面内または短いスクロールで使えるか。
4. 状態、色、アイコン、文言が矛盾していないか。
5. 権限拒否や非対応端末でも行き止まりにならないか。
6. チュートリアルの全課題が、指示どおりの操作で進むか。
7. 実験結果より設定や説明が目立っていないか。
