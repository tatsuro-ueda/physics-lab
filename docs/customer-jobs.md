# Customer Jobs：学校端末で物理実験を行う理科教師

## この文書の位置づけ

本書は、ボブ・マエスタのJobs to Be Doneの考え方に沿って、先生がタブレット物理実験ラボを「雇う」状況と、求める進歩を記述する。

ただし、参照したllm-wikiには「中学校・高校の理科教師が、生徒の端末で物理量を計測する場面」の直接証拠がない。したがって、本書では次の証拠レベルを混ぜない。

- **隣接証拠**：小学校教員のインタビュー、授業観察、学校端末運用など、リモートwikiに記録された事実
- **設計仮説**：加速度・音響ストップウォッチの開発から導いた、今後検証すべき仮説
- **未検証**：中高教師本人へのインタビューや授業観察が必要な事項

## 主人公

授業で生徒の学校端末を使い、加速度や時間などの物理量を計測させたい理科教師。

担当校、校種、教員経験、端末環境、授業内での端末利用権限は一定ではない。本書は単一のペルソナ像を作るのではなく、先生が特定の状況で達成しようとする進歩を扱う。

## Job Statement

> 生徒に物理現象を実感させたい授業で、限られた準備時間と学校端末の制約の中でも、全員が迷わず計測を始められるようにしたい。そうすれば、端末操作の説明ではなく、予想・実験・結果の解釈に授業時間を使える。

このJob Statementは**設計仮説**である。学校現場の端末制約と実験運用負荷には隣接証拠があるが、中高教師本人による表現ではない。

## 望む進歩

### 機能的な進歩

- **設計仮説**：QRまたはURLから開き、インストールなしで計測を始めたい。
- **設計仮説**：iPadやChromebookなど端末が違っても、同じ授業手順で使いたい。
- **設計仮説**：専用実験器具の配布・回収・設定を増やさず、端末内蔵センサーを使いたい。
- **設計仮説**：生徒が自分で開始、観察、停止、値の読み取りまで進められるようにしたい。
- **設計仮説**：計測が失敗した班に、教師が付ききりにならず回復方法を示したい。
- **設計仮説**：授業で扱う物理量、単位、軸の向きを画面と一致させたい。

### 感情的な進歩

- **設計仮説**：端末トラブルで授業が止まる不安を減らしたい。
- **設計仮説**：全班の操作状況を常に監視しなくても、授業を進められる安心感がほしい。
- **設計仮説**：新しい道具を使うこと自体ではなく、物理を理解させられた手応えを得たい。
- **隣接証拠**：班ごとに進行が異なる実験では、教師の統率・安全確認負荷が高い。

### 社会的な進歩

- **設計仮説**：他の先生にも短く説明でき、同じ手順で再現してもらいたい。
- **設計仮説**：高価な器具がなくても、学習目標を損なわない授業として説明したい。
- **隣接証拠**：ツールの普及には、研修だけでなく端末からすぐ開けることや同僚間の自然な伝播が影響する。

## 進歩を求める状況

### 授業を計画するとき

- 生徒にグラフと実際の動きを結びつけさせたい。
- 専用器具の数、準備時間、端末環境が授業案を制約する。
- **隣接証拠**：高価な物理実験器具を、安価で準備の少ない方法へ置き換えた実践がある。
- **未検証**：中高の物理実験で、器具不足がどの程度授業選択を変えているか。

### 授業を始めるとき

- 端末の配布、ログイン、URL入力、権限許可に時間を使いたくない。
- 生徒が別々の場所で迷うと、一斉説明を何度も繰り返す必要がある。
- **隣接証拠**：学校採用端末はiPad、Chromebook、Windowsなど多様で、端末利用ルールには地域差がある。
- **設計仮説**：初回ツアーと体験チュートリアルが、教師の口頭説明を減らす。

### 実験中

- 全班が安全に、同じ学習目標へ向かっているか把握したい。
- 操作質問ではなく、現象の違いや予想について生徒と話したい。
- **隣接証拠**：班別実験では進め方が異なり、全班の安全確認をリアルタイムで行う統率負荷が高い。
- **設計仮説**：状態表示と失敗時の次行動が明確なら、操作支援の呼び出しを減らせる。

### 授業後

- 次回も同じURLと手順で使いたい。
- 同僚へ授業方法を渡したい。
- **隣接証拠**：同僚間の伝播と端末上の見つけやすさがツール普及を助ける。
- **未検証**：計測データの保存や提出が、先生にどの程度必要とされるか。

## 採用を動かす四つの力

ボブ・マエスタ式に、現在の方法から新しい方法へ切り替わる力を整理する。

```text
現在の方法                              新しい方法
┌────────────────┐                ┌────────────────┐
│ Habit: 慣れた器具・演示・口頭説明 │                │ Pull: すぐ開く・自分で進める │
│ Anxiety: 権限・端末差・授業停止   │                │ Push: 準備・統率・器具の負荷 │
└────────────────┘                └────────────────┘
```

### Push：現状から押し出す力

- **隣接証拠**：班別実験の進行差により、教師の統率・安全確認負荷が高い。
- **隣接証拠**：安全記録を十分に残そうとすると記録負担が増える。
- **隣接証拠**：高価な実験器具を低コストな代替方法へ置き換える需要がある。
- **設計仮説**：専用器具の準備、配布、回収、故障対応が授業導入の障害になる。
- **設計仮説**：端末操作の一斉説明が、予想や考察に使える時間を減らす。

### Pull：新しい解決策へ引く力

- **設計仮説**：ブラウザで開くだけで、追加インストールなしに使える。
- **設計仮説**：生徒の手元にある端末を計測器として使える。
- **設計仮説**：初回ツアーが画面の場所を、チュートリアルが実験手順を案内する。
- **設計仮説**：値とグラフをその場で見て、動きと物理量を結びつけられる。
- **設計仮説**：共通UIにより、別の計測ページでも学んだ操作を再利用できる。
- **隣接証拠**：端末からすぐ開ける導線と同僚間の伝播は、利用の広がりを助ける。

### Anxiety：採用を妨げる不安

- **隣接証拠**：学校端末にはセキュリティ制約があり、校外から開けないデータもある。
- **隣接証拠**：学校や地域によって端末利用ルールが異なる。
- **隣接証拠**：教員のスマートフォン利用制限が強まり、授業利用にはWeb版が適する。
- **設計仮説**：センサー非搭載、権限拒否、OS差で一部の端末だけ動かない不安がある。
- **設計仮説**：授業中にネットワークや外部CDNが止まり、全員が使えなくなる不安がある。
- **設計仮説**：計測値が教科書や既存器具と一致しない不安がある。
- **設計仮説**：端末操作へ集中し、生徒が物理現象を見なくなる不安がある。

### Habit：現在の方法へ留める習慣

- **設計仮説**：教師が前で演示し、生徒は観察する授業なら進行を統制しやすい。
- **設計仮説**：使い慣れた専用器具は、準備手順と故障時の対処が分かっている。
- **設計仮説**：紙の手順書と一斉説明は、端末環境に左右されにくい。
- **設計仮説**：新しいアプリを事前確認する時間がなければ、昨年と同じ方法が選ばれる。
- **隣接証拠**：普及は正式研修だけで決まらず、端末上の導線や同僚からの伝播にも左右される。

## 競合する解決策

Jobの競合は同種アプリだけではない。先生が同じ進歩のために選べる方法を含む。

| 競合 | 雇う理由 | 解雇する理由の仮説 |
|---|---|---|
| 専用実験器具 | 測定対象が明確で、授業実績がある | 台数、費用、準備、保管が必要 |
| 教師による演示 | 進行と安全を統制しやすい | 生徒自身の操作と試行が少ない |
| 動画・シミュレーション | 失敗せず短時間で見せられる | 現実のばらつきや自分の計測にならない |
| 紙の記録・手動ストップウォッチ | 端末権限やAPIに依存しない | グラフ化や反復計測に手間がかかる |
| 市販のセンサーアプリ | 機能が多く、既知の利用例がある | インストール、広告、端末差、説明量が増える可能性 |
| 安価な自作器具 | 低コストで現象を示せる | 生徒端末による定量計測とは役割が異なる |

「解雇する理由」は、wikiの直接証拠ではなく**未検証の比較仮説**である。

## 採用タイムライン

### 1. 最初の気づき

既存の器具や授業方法に、費用、準備、統率、生徒の主体性などの不満を感じる。

- 根拠：実験統率負荷と低コスト代替には隣接証拠がある。
- 未検証：本アプリを探し始める具体的な出来事。

### 2. 受け身の探索

研修、同僚、教材共有、検索、QRなどで候補を知る。

- 根拠：端末上の導線と同僚間の伝播には隣接証拠がある。
- 設計仮説：説明を読まなくても試せる公開URLが探索を前へ進める。

### 3. 積極的な比較

自分の端末で開き、権限、計測値、授業手順、失敗時の回復を確認する。

- 設計仮説：インストール不要、初回ツアー、HTTPS、端末別エラーが比較時の不安を減らす。

### 4. 最初の利用

教師が演示または小規模な班で試し、授業時間内に成立するかを見る。

- 未検証：最初から全員利用するか、小規模試行から始めるか。

### 5. 継続利用

同じURLと操作体系で別の単元にも使い、同僚へ共有する。

- 設計仮説：ページ間で共通するヘッダー、操作、チュートリアルが再学習を減らす。
- 隣接証拠：同僚間の自然な伝播が普及を助ける。

## プロダクト設計への含意

1. インストールではなく、HTTPSのURL・QRから開始できることを守る。
2. 授業中の外部ネットワーク依存をなくす。
3. iPadとChromebookを主要な検証対象にする。
4. 初回ツアーは場所案内、チュートリアルは体験と役割を分ける。
5. 生徒だけで失敗から回復できる短い案内を置く。
6. ページ間で開始・停止・削除・チュートリアルの位置と言葉を揃える。
7. 物理量、単位、正方向を画面上で確認できるようにする。
8. センサー・マイクを使えない端末でも、原因と代替行動が分かるようにする。
9. 教師が30秒程度で用途を判断できる画面にする。
10. 中高教師への一次調査が得られるまで、保存・提出・授業管理機能を必須Jobと断定しない。

## Boundary Note

wikiにない次の事実は、本書で確定事項として扱わない。

- 中高理科教師が生徒端末計測を採用・断念した具体的な出来事
- phyphoxなど既存アプリへの満足・不満
- 中高物理実験でのICT機器整備状況
- データ保存・提出・授業管理の必要度
- 教師が許容できる準備時間、説明時間、故障率

一次証拠を得る候補として、科教協の中高部会の先生へのインタビューと授業観察が挙げられる。新しい証拠を得たら、設計仮説を採用・修正・棄却し、Confidenceを更新する。
